## Requirements

- **Node 20+** *or* [**Bun**](https://bun.sh) **1.3+**: either runtime works
- A POSIX shell (Linux, macOS, or WSL)

The published npm package ships a prebuilt `dist/`, so it runs on plain Node out of the box, no Bun required. Installing Bun is optional but recommended for *developing* Harnery: Bun runs the TypeScript source directly (zero build step), so edits are live. The package's `exports` resolve to TS source under Bun and to the built `dist/` everywhere else, so the two stay in lockstep.

:::note[One Bun-only command]
`harn tunnel` is the single exception: its reverse-proxy gate worker is a `Bun.serve` process, so that one command requires Bun installed. It fails with a clear message on a Bun-free host. Everything else runs on Node.
:::

## Install

<Tabs>
  <TabItem label="Global">
    ```bash
    npm install -g harnery
    harn --help
    ```
  </TabItem>
  <TabItem label="Project">
    ```bash
    npm install harnery
    npx harn --help
    ```
  </TabItem>
  <TabItem label="bun">
    ```bash
    bun add -g harnery
    harn --help
    ```
  </TabItem>
</Tabs>

## npm vs git clone: what each gives you

`npm install harnery` ships the **CLI + coord engine** (`bin`, `dist`, `src`, `schemas`): everything you need to compose Harnery into a CLI, wire hooks with [`harn init`](/cli/init/), and run the coord layer. It does **not** include the `web/` dashboard or this `docs/` site; those are heavy app/build trees that live in the git repo only.

To run the dashboard ([`harn web up`](/cli/web/)), `git clone` the repo and launch it from there, pointing at your project with `--coord-root <dir>` or running it from inside the project. (`harn web up` prints these steps if invoked without the clone present.)

## From a git clone (one step)

Cloned the repo instead of installing from npm? One script does everything: it installs deps, builds the Node `dist/` (skipped when Bun is present), runs [`harn init`](/cli/init/) to wire your project, and links the bins onto your `PATH`:

```bash
bash harnery/install.sh
```

Run it from your project (it auto-detects the project root via git, falling back to Harnery's parent directory). It's idempotent, so it's safe to re-run. Flags: `--project-root <dir>`, `--link-dir <dir>` (default `~/.local/bin`), `--no-link`, `--harness <id>`.

To reverse it, run the mirror script — `bash harnery/uninstall.sh` (see [Uninstall](#uninstall) below).

## First-run check

```bash
harn doctor
```

This verifies the runtime + dependencies Harnery expects (Node version, optional Python, optional restic for backup, etc.). It exits 0 if everything's wired and prints OS-specific install instructions for anything missing.

## Bootstrap a project

```bash
harn init
```

Run once per project to create the `.harnery/` coord root and wire the harness hooks. Auto-wires all three harnesses; pass `--harness claude-code` (default), `cursor`, or `codex`. Idempotent and non-destructive: safe to re-run, and it preserves hooks you already have. See [`harn init`](/cli/init/) for `--dry-run` and details.

## Uninstall

Reverse what `harn init` (or `install.sh`) wired in — one command. Your `.harnery/` history is kept by default.

<Tabs>
  <TabItem label="npm">
    ```bash
    harn uninstall      # unwire the harness hooks (keeps .harnery/)
    npm rm -g harnery   # remove the engine + CLI
    ```
  </TabItem>
  <TabItem label="git clone">
    ```bash
    bash harnery/uninstall.sh   # mirror of install.sh: unwire + unlink the bins
    ```
  </TabItem>
</Tabs>

Pass `--purge-state` to either to also delete the `.harnery/` coord root (session history, councils, identities) — destructive. The clone-path script also takes `--link-dir`, `--keep-links`, `--project-root`, `--harness`, and `--dry-run`. See [`harn uninstall`](/cli/uninstall/) for the full flag set and the surgical, hook-preserving behavior.
