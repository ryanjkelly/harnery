```
harn devtools [--tool <name>] [--usage] [--window-days <n>] [--format table|json]
```

Reports the on-disk state of the AI coding agents Harnery is built for — Claude
Code (`~/.claude`), Codex (`~/.codex`), and Cursor (`~/.cursor`) — in one place:
logged-in status, plan / seat tier, auth expiry, session counts, and, where the
tool records them locally, rate-limit / quota windows.

The default report comes from files on disk — no network call. Auth tokens are
inspected only for their non-secret claims (email, plan, expiry); the token
strings themselves are never printed. A signal a tool keeps server-side surfaces
as blank with a note rather than a guess.

One opt-in enrichment reaches the network: for Cursor, `devtools` reads the IDE's
own session token from `state.vscdb` and calls cursor.com's dashboard API — the
same request Cursor's UI makes for its Spending page — to fill the billing-cycle
countdown and total/API/first-party percent-used. It's the user's own token
reading the user's own data, and `--no-api` turns it off.

## What each tool exposes locally

| Signal | Claude Code | Codex | Cursor |
| --- | --- | --- | --- |
| Logged-in | ✅ credential + expiry | ✅ `auth.json` | ✅ `state.vscdb` |
| Account | ✅ email | ✅ email (JWT claim) | ✅ email (`state.vscdb`) |
| Plan / tier | ✅ seat tier | ✅ live plan (rate-limit snapshot) | ✅ Stripe membership + status |
| Auth expiry | ✅ | ✅ | ❌ |
| Session count | ✅ transcripts | ✅ threads (`state_5.sqlite`) | ✅ chats (`composerHeaders`) |
| Quota / reset windows | ❌ (live via `/usage`) | ✅ 5h + weekly, from `token_count` | ✅ billing cycle, via session token |
| Percent used | ❌ (live via `/usage`) | ✅ 5h + weekly | ✅ total / API / first-party |
| Token totals (`--usage`) | ✅ transcript scan | ✅ best-effort | ❌ server-side |

Cursor's account, plan (`stripeMembershipType`), and subscription status come
from its IDE state DB (`state.vscdb`, a SQLite file). That read needs a SQLite
engine — under Bun (`bun:sqlite`) it works; on a plain-Node host it degrades to
login-presence only. On WSL the DB lives on the Windows side and is usually
WAL-locked while Cursor runs, so the reader opens it immutable rather than copying.

Cursor's **usage + billing** (percent used, billing-cycle end, on-demand spend
cap) are not on disk — they live on `cursor.com`. The enrichment step fetches
them with the IDE's own session token, so the card still shows them without any
key to mint. Two gaps remain: **exact token totals** aren't exposed by that
endpoint, and Claude Code's own quota is still live-only (via `/usage`).

## Options

| Flag | Effect |
| --- | --- |
| `--tool <name>` | Restrict to a tool (repeatable): `claude-code`, `codex`, `cursor`. Default: all three. |
| `--usage` | Also scan local transcripts for approximate token totals. Opt-in because transcripts can be gigabytes. |
| `--window-days <n>` | With `--usage`, only count transcripts modified within N days (default 7). |
| `--no-api` | Skip the Cursor API enrichment even when a key is configured (stay fully local). |
| `--format <type>` | `table` (default) or `json`. |

## Cursor enrichment

Two independent network calls run during enrichment (both skipped by `--no-api`):

**Usage + billing (no key needed).** `devtools` reads Cursor's own session token
from `state.vscdb` and calls `cursor.com/api/dashboard/get-current-period-usage`
— the same request the Spending page makes — authenticating with the WorkOS
session cookie the IDE already holds. The Cursor entry gains `usage: { cycleStart,
cycleEnd, totalPercentUsed, apiPercentUsed, firstPartyPercentUsed,
includedLimitCents, spendLimitCents, spendUsedCents }`. An expired token is
skipped rather than called.

**Cloud Agent activity (needs an API key).** If you store a Cursor API key, the
entry also gains `api: { ok, keyName, cloudAgents }` from the public `/v0` API:

```
harn devtools cursor-key set          # paste the key on stdin
harn devtools cursor-key status       # is a key configured, and from where
harn devtools cursor-key clear        # remove it
```

The key is read from `CURSOR_API_KEY` first, then `~/.config/harnery/cursor-api-key`
(stored `0600`). Cursor's public usage/spend API is **Team-only** (`/teams/*`), so
an individual Pro/Pro+ key returns no token cost there — the key path only adds
Cloud Agent runs (`/v0/me`, `/v0/agents`). The richer billing numbers come from
the session-token path above, not the key.

## Examples

Default report:

```bash
harn devtools
```

```
── codex ───────────────────────────────────
   logged in    yes
   account      dev@example.com
   plan         team
   auth expires 2026-07-07T08:28:13Z
   sessions     36
   quota (5h)   21% used · resets 2026-05-09T09:24:06Z
   quota (weekly)   43% used · resets 2026-05-12T06:01:50Z
```

Token totals for the last day, JSON, one tool:

```bash
harn devtools --tool claude-code --usage --window-days 1 --format json
```

The JSON envelope is `{ ok, generatedAt, windowDays, tools: ToolStatus[] }`,
where each `ToolStatus` carries `installed`, `loggedIn`, `account`, `plan`,
`rateLimitTier`, `authExpiresAt`, `sessions`, `lastActivity`, `quota[]`,
`tokensUsed`, `api`, `usage` (Cursor billing/percent-used), and `notes[]`.
